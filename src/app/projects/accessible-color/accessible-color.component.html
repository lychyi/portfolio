A walkthrough of how we designed a color system at Illumina that put accessibility at the forefront while serving our vast ecosystem of products in a consistent manner.

A color system for the web is somewhat unique. Color conveys important information and provides context (warnings, errors, and successes). Moreover, it reflects your product's brand and can even be an important part of the brand in it of itself. Surprisingly though, color is often times inconsistent across user interfaces and treated as an afterthought. Without guidelines, does it matter if we're using this orange or this orange? Furthermore, how do we even know when and where to use colors. Can you can read this orange on this background? If you have monochromacy (a type of color blindness where you see the world in only one frequency of color or in blacka and white), it may be difficult since the relative luminance values of each color may be too close, resulting in low contrast. Can you read this: "I may be difficult to read on certain devices."? You can probably make it out but it isn't easy and this is what a person with monochromacy will encounter with the orange shown earlier against this background.

Accessibility matters.

At Illumina, we faced challenges like this. Our brand colors were never designed for software user interfaces on the web, let alone an entire family and ecosystem of products. That orange color we presented earlier? That's Illumina Pumpkin Orange. Our flagship brand color. When we use that on software user interfaces with a light-grey background, we're essentially making accessibility difficult. I'm not saying anyone advised software developers to use Illumina Pumpkin Orange on grey but without guidelines and principles (and some unspoken rule that all scientific software needs to have a light-grey or white background), you can see how it is easy to misuse.

Inaccessible and inconsistent

Without a formal color system, software at Illumina could be designed with any palette. Color was up to the designer or developer to implement. This resulted in colors that were inconsistent across user interfaces and didn't necessarily reflect our brand or guarantee accessibility for all of our users.

Creating a color system

So how does one design a color system? If you're a company or brand that has many software products, like Illumina, you'll want to a decent-sized color palette to choose from. We didn't want one so big that you could choose just about anything. We didn't want one so restrictive that all user interfaces would look nearly identical. We also wanted guidelines and principles to be baked in so that the user of the color system would have an easier time reasoning about it.

With that in mind, we decided to develop a diverse color system specifically for the web with colors complementary to our brand and at the same time, promote standardized accessibility.

Beg, Borrow, and Steal

Luckily, the hard part was already done by someone else.

Color theory and systems for web user interfaces have been solved before. When our design team met up to discuss color, it immediately became clear that it was difficult to solve this organically. We needed a programmatic, methodical way to go about this.

Color is the way which we perceive segments of the visible light spectrum. We're talking about wavelengths, contrast, luminance, hue, saturation, lightness, etc. In other words, functional parameters of a complex physics equation. We have a natural tendency to attach emotion and subjectiveness to color, but a solid color system needed to be based upon color theory and a bit of math.

Luckily, the hard part was already done by someone else. The IBM Design Language had already neatly solved this problem in an elegant way. They had a color system with baked in accessibility and it had eough variation to give us a diverse palette. We made a few small modifications but this color system unashamedly borrows from them.

Thank you IBM Design.

Accessibility comes down to contrast. Contrast is math. Make the math simple.

Accessibility really just comes down to contrast. Your eyes are good at distinguishing the contrast between colors unless you have some form of color blindness. In order for us to use colors in an accessible way, we have to assume that the text that conveys information in our user interfaces do so with enough contrast regardless of color. There are minimum contrast ratios prescribed by the W3C that define what makes text legible for all users. The equation is based on relative luminance. Once you get relative luminance, you plug it into this equation: (L1 + 0.05) / (L2 + 0.05). Where L1 is the lighter of the two colors and L2 is the darker of the two colors.

But we're busy. Having to plug colors into an equation to verify contrast for accessibility is tedious at best. What's more problematic is that it wouldn't do us any favors when designing for accessibility.

Color and numbers?

Pick primary hues to start

What we ended up doing at Illumina was picking a few starting hues for our color palette. This is the largely subjective part and arguable the most important since it sets us apart from just outright using someone else's color palette. The colors were inspired by the vibrant optical filters inside of our Illumina sequencers. We also wanted boldness and vibrancy to be the dominant theme for our colors so this was perfect.

Generate shades from those hues

We then generated a variety of shades (we did this programmatically using chroma.js, a very handy JavaScript library when working with color) from these hues based on relative luminance and gave it a numerical label. This label corresponds to the contrast calculation we did in the example above. Whenever two colors have a difference of 50 or greater, we can be assured that there is enough contrast by WCAG standards to be accessible.

Voila. Consistent, Accessible Colors

This results in a color system that is diverse yet consistent. We carefully selected colors that reflect our brand and generated enough variation for our current user interfaces and the ones to come.

Subjectivity meets practicality

A successful color system needs to strike the right balance between form and function.

Programmatic color generation does have some issues. It doesn't account for how we perceive color as humans. For example, our 'warning' (yellow) hue started getting muddy from shades 40 and beyond. Unfortunately, this is just the nature of how orange and yellow hues get when their shades get darker so we just nixed shades 40 through 90. Another example is that we removed the darkest shade (it was nearly black) and added a lighter shade because we felt that our user interfaces needed more subdued versions of the hues.

A successful color system needs to strike the right balance between form and function. Our aim was to craft a system of colors that really reflected a consistent brand while promoting accessibility at it's core. We were able to accomplish this while still making it beautiful.
