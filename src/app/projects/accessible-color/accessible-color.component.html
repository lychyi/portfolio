Illumina's software family is a diverse ecosystem of products that run the gamut of function and form. When I joined the company, I joined a cross-functional team dedicated to developing a color system for web user interfaces. Our challenge was to create a color system that was easy to understand and use, yet flexible enough to use throughout the organization.

A color system for the web is somewhat unique. Color can convey important information and while providing helpful context like "success" or "error". Moreover, it reflects your product's brand and can even play a critical role of promoting it (Tiffany blue for example). But perhaps unsurprisingly, color doesn't get the attention it deserves when designing software and relating it to the brand. I mean, it's purely subjective right? Without guidelines, does it matter if we're using this orange or this orange? How do we know when to use what colors for what function? Can you can read this orange (#FFB441 in hex) on this background? If you have monochromacy (a type of color blindness), it may be difficult. Can you read the following? "This is what a person with monochromacy would see if viewing #FFB441." Probably, but you're struggling because of the contrast. Yet this is similar to what a person with monochromacy will encounter if we don't think about accessibility when choosing our colors.

Accessibility matters.

At Illumina, we faced challenges like this. Our brand colors were never designed for software user interfaces on the web, let alone an entire family and ecosystem of products. That orange color we presented earlier? That's Illumina Pumpkin Orange, our flagship brand color. When we use that on software user interfaces with a light-grey background (like the one you're viewing now), we're essentially making life difficult for our color blind users. And being our company's main brand color, you can easily see how it can be misused.

Inaccessible and inconsistent

Without a formal color system, software at Illumina could be designed with any color. It was up to the designer or developer and perhaps whatever mood they were in. And if you take a look at Illumina's history, you can see this in our products: inconsistent color choice and poor accessibility for all of our users.

Creating a color system

With branding and our users in mind, we decided to develop a diverse color system specifically for the web with colors complementary to our brand and at the same time, promote standardized accessibility.

Beg, Borrow, and Steal

Luckily, the hard part was already done by someone else.

Color theory and systems for web user interfaces have been reasoned about before. When our team met up to discuss our approach, it immediately became clear that it was difficult to solve this in a vaccum. We needed some inspiration.

Color is the way which we perceive segments of the visible light spectrum. We're talking about things like wavelengths, contrast, luminance, hue, saturation, lightness, etc. In other words, the output of a complex equations. We have a natural tendency to associate emotion and subjectiveness to color, but a solid color system needs to be based upon color theory and a bit of math.

Luckily, the hard part was already done by someone else. The IBM Design Language had neatly solved this problem. They had a color system with baked in accessibility and it had enough variation for a diverse palette. When it was all said and done, we made a few modifications but this color system unashamedly borrows from them.

Thank you IBM Design.

Accessibility comes down to contrast. Contrast is math. Make the math simple.

How do we create colors that help our color blind users? Even if you have color blindness, your eyes can distinguish between different contrast points. In order for us to use colors in an that way, we have to assume that the text that conveys information in our user interfaces do so with enough contrast regardless of color we choose. There are minimum contrast ratios prescribed by the W3C that define what makes text legible for all users. The equation is based on relative luminance. Once you get relative luminance, you plug it into this equation: (L1 + 0.05) / (L2 + 0.05). Where L1 is the lighter of the two colors and L2 is the darker of the two colors.

Having to plug colors into an equation to verify contrast for accessibility is tedious at best. In addition, making it difficult wouldn't do us any favors if we're trying to promote the importance of designing for accessibility.

An easy way to solve this is to simplify the math. We labeled each color in the system with a number. This number corresponds to a specific relative luminance value and a designer or developer can determine what is appropriate contrast by simple subtraction.

How we built this.

Pick primary hues to start

We started with some starting hues for our color palette. This is the largely subjective part and arguable the most important part in distinguishing our brand from another. The colors were inspired by the vibrant optical filters inside of our Illumina sequencers.

Generate shades from those hues

We then generated a variety of shades (we did this programmatically using chroma.js) from these hues based on relative luminance and gave it a numerical label. This label corresponds to the contrast calculation we did in the example above. Whenever two colors have a difference of 50 or greater, we can be assured that there is enough contrast by WCAG standards to be accessible.
