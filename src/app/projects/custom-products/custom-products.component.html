As the lead designer and front end developer, I was responsible for designing a brand new experience and engineer it from the ground up.

The microarray assay has been a core technology of Illumina for nearly a 15 years. It has been utilized to support some of the most well-known genomics entities out there like 23 and Me and Ancestry.com. Creating customized versions of this assay is still the heart of genomic research in many areas today. In 2016, we were given the green light to redesign the software that creates those custom assays. We really wanted to modernize it and fix workflow issues that the old one had.

Out with the old...

The old software application that allowed our users to create custom microarray assays was dubbed the "Assay Design Tool" or ADT for short. While it wasn't a very creative name, it certainly got the job done even with some usability and experience hiccups along the way. The messaging wasn't clear and there was a hitch in the workflow so that customers would have to submit their inputs twice in order to have a working design.

... in with the new

Our new software had to accomplish a few goals:

  1. It should have a similar look & experience with DesignStudio, a sister application that does the same thing but with our newer NGS technology assays yet...
  2. It shouldn't be radically different from ADT, we didn't want to alienate current users familiar with it.
  3. Finally, it should be modernized and use the latest visual design trends and interactions that users expect. We were going to be launching this software with our latest and greatest microarray product so it should match in terms of innovation.

A good time to use new front end technology

ADT was developed nearly 10 years ago so we really wanted to start the front end from scratch. It was dated and not really worth saving from a code reuse standpoint.

What ADT used to look like.

During development, Angular was about to release version 1.5 and the popular framework was at its most mature state. The supporting tools had gotten really good and applications practically bundled and deployed themselves (of course you still had to configure things like Webpack which was hard enough but totally worth it). Moreover, the user interface needed a facelift and we could easily solve common painpoints of ADT just by re-thinking the end-to-end workflow.

The journey took a about 6 months but ultimately, the results were great. We evolved the platform from a simple forms-based web page to an elegant, single-page Angular application. While delivering everything we committed to on time.

How it was built

The importance of component-driven design

By simply addressing the design elements from an atomic level and then assembling upwards, more complex interfaces can take shape with predictable results.

Component libraries pay for themselves over time

For just about any project that I start, I spend time to craft a library of quality 'atoms' that are used to prototype user interfaces with Goldilocks-fidelity as soon as possible (not too high, not too low). I do this for two reasons:

  1. User feedback on believable prototypes are generally more authentic compared to when users know they're just testing a 'prototype'.
  2. Validated features and screens can easily be passed onto developers with little to no modifications since they are visually true to the final design.

Don't get me wrong, I love pen and paper and wireframing for quick ideation at the beginning of the process, but those are a one-time use effort. Having my own well-crafted component library is something that has multiple uses and gives value throughout the lifecycle of a project. By simply addressing the design elements from an atomic level and then assembling upwards, more complex interfaces can take shape with predictable results.

DesignStudio's Component library

Sketch is built for the job

Sketch has made it elegant to build component libraries with their symbols feature. It allows easy reuse of common user interface elements across artboards and fits perfectly with a component-driven design process. Ultimately, it makes the design process fast, often times turning ideas into prototypes within the same day (or hour).

One vastly underrated feature of Sketch is the ecosystem of plugins that the community has built around it. Plugins like Runner and Craft really show how creative you can get with them and how a bit of code can save designers hours every single day. These plugins also help keep my designs component-driven. They force me to be conscious about building useful, re-usable components and often times, this translates into re-usable CSS styles and JavaScript code too.

Designing for developers

We can apply design to developers too. Setting up a new AngularJS project is one of the most difficult things to get right. Choosing your build tools and writing npm scripts are the backbone of setting up any project. The choices that you make here will result in a more productive and pleasant developer experience.

JavaScript. Love it or loathe it.

For this project, I combined BrowserSync for quick cross-browser testing along with Gulp to detect code changes and run tasks. When we were ready to start deploying to our environments, I wrote various configurable deployment scripts for our different environments and stashed them in npm. No more putzing around with config transforms, we built for all of our environments in one go so we can have them on-hand when ready. Finally, I configured TeamCity to build NuGet packages so that Octopus could deploy them when ready (we weren't deploying ASP.NET packages but it was something familiar to our DevOps team and it could deliver packaged file systems just fine).

Webpack, Optimizations, and Performance

This new product was a great time to focus on front end performance. We were able to reduce our app deployment size to about 1.4MB (we started around 4MB) after using Webpack and innovative loaders like bootstrap-loader. Webpack was also able to save us a few bytes here and there by removing deadend code, unused functions, and documentation all before we were uglifying and compressing. Finally, all of the assets were optimized according to their file type and we made smart design choices about the type of images we would use and where we would use them.
